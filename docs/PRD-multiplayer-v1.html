Accounts, Stories, and the Host's Table

Turning the solo character sheet into a shared table — sign-in, campaigns, and a Host who can see their players.

Status — draft for your review Date — 2026-07-11 Scope — the multiplayer phase Decides nothing final — everything here is up for change
01

The goal

Today Vertex is a proof of concept: one person, one browser, characters saved on the device (with an optional Google Drive backup). This phase makes it functional for a real group. People sign in. A Host creates a Story — the campaign — and shares an invite code. Players join with their one character. The Host opens the Story and sees the whole party. Everyone's data lives in the cloud so it follows them across devices.

The organizing idea is the Story: a container for a group of players, their characters, and — as later phases land — the Host's own material (NPCs, items, settings, notes, and the Fate/arc tracking Vertex runs on). A Player sees their character and a light party roster; the Host sees everything in their Story.

03

The pieces, named

Account

A person. Signs in with Google or email + password. Owns characters; joins or hosts Stories. Not "a Host" or "a Player" globally — that's decided per Story.

Story

The campaign. Has one Host, a roster of Players, their characters, an invite code, and (later) the Host's NPCs, items, settings, and notes.

Membership

The link between an Account and a Story, carrying the role (host / player) and which character they brought.

Character

Owned by one Account. Can be attached to any number of Stories at once (usually one). The existing sheet, now in the cloud.

Invite code

A short code/link (e.g. VTX-7QK2) a Host shares. Entering it makes you a Player in that Story.

Guest

Using Vertex without signing in — a local-only character, fully playable offline. Can be claimed into an account later.

05

The core flows (v1)

  1. Sign in — or don't. Continue as a guest (local only), or sign in with Google or email + password.
  2. First sign-in brings your characters. Any characters already on this device (or in Drive) are auto-imported into your account as your personal characters.
  3. Host: create a Story. Name it; you become its Host and get an invite code to share.
  4. Player: join with a code. Paste the code, pick which of your characters to bring (your one active character for that Story).
  5. Host: open the Story. See the roster of players and, for each, open their current character sheet (read-only snapshot of their last save).
  6. Player: play. Manage your own sheet as today; glance at the party roster to anchor Tethers and Echoes.
06

Data model

Relational, which is why a SQL backend fits. Five core tables for the MVP; the rest arrive with their phase.

users

id · email · display_name · created_at — provided by the auth service.

stories

id · name · host_id → users · invite_code · created_at

memberships

id · story_id · user_id · role (host|player) · character_id · joined_at

characters

id · owner_id → users · data (sheet JSON) · updated_at — no story link here; characters reach Stories through memberships (many-to-many).

invites

id · story_id · code · created_by · expires_at · used_by · revoked — one-time, 24h, host-revocable.

later: story_content

npcs · items · notes/settings · fate_log — each keyed to a story_id, Host-only.

Reuses today's workA character's sheet stays the same JSON the app already produces. characters.data holds it — so the rules engine, rendering, and creation wizard don't change. Only the storage seam (Vertex.storage) swaps from localStorage to the cloud, exactly as the architecture anticipated.
07

Backend recommendation — Supabase

You asked for a managed service on a free tier, and for me to recommend one. I'd choose Supabase.

  • The access rules are the hard part, and Supabase makes them clean. Its Row-Level Security lets us write a rule like "a user may read a character if they own it, or if they host the Story it belongs to" directly on the data. That per-Story, role-based access is exactly our requirement.
  • Auth is built in — Google and email + password out of the box, so no passwords ever pass through our code.
  • The model is relational (users ↔ stories ↔ memberships ↔ characters), which Supabase's Postgres expresses naturally.
  • Real-time is included for the later live-Host-view and Echo/Sacrifice phases — no re-platforming.
  • Free tier comfortably covers a project at this stage; it scales if it ever needs to.

Alternative — Firebase. Also free-tier and turnkey, with the tightest Google sign-in. I'd pass on it here mainly because its security rules get awkward for "is this user the Host of this character's Story" checks — the one thing we most need to get right.

What this means for youYou would not run a server. Setup is a dashboard plus a small amount of glue code behind Vertex.storage. The guest/local path stays, so the app still opens and works with no account.
08

Phased roadmap

Phase 0 — Foundation

Accounts & the cloud seam

Get identity and storage in place without changing how the sheet plays.

  • Supabase project; users / characters tables.
  • Sign-in: Google + email & password; sign-out; guest mode preserved.
  • Cloud storage adapter behind Vertex.storage; a character syncs to the account.
  • Auto-import local/Drive characters on first sign-in.
Phase 1 — MVP multiplayer the review target

Stories, joining, and the Host's roster

The functional core you described: link characters to a Story; the Host sees the players.

  • Create a Story → become Host → invite code.
  • Join via code → become Player → attach your one active character.
  • memberships + role-based access rules enforced in the database.
  • Host screen: party roster + open any player's character (read-only snapshot).
  • Player screen: own sheet (as today) + a light party roster.
Phase 2 — The Host runs the table

Host content & live view

Give the Host the tools Vertex actually needs at the table.

  • Host-only Story content: NPCs, item/loot library, settings & notes, per-character Throughline.
  • Fate / arc tracking: award Fate, mark Drift, flag Crossings.
  • Real-time Host view (see player edits live) and light presence.
  • Host acts on player sheets — either direct edit or "Host suggests → player applies" (to be decided).
Phase 3 — The living table

Player-to-player & introspection

The most Vertex-specific social mechanics.

  • Echo (move Fate between players) and Sacrifice.
  • Vignette flows; the Crossing as a guided moment.
  • Notifications / invites by email; richer roster (shared Tethers).
09

Migration & the guest path

  • Nobody loses a character. On first sign-in, characters already on the device (and any in Drive) are copied into the new account automatically.
  • Guest mode stays. The app still opens with no login and works offline on a local character — then offers "claim this into an account" when they're ready.
  • Google Drive's role. Once the cloud is primary, Drive sync becomes redundant for multiplayer. Open question below: retire it, or keep it as an optional personal export.

11

Not in v1 · risks

Deliberately out of the MVP

Real-time Host view · Host editing player sheets · Echo / Sacrifice automation · NPC & item management · Fate/arc tooling · notifications · a native mobile app · spectators.

Risks to watch

  • Access-rule correctness. A misconfigured rule that lets a player read another's sheet is the worst failure mode — it needs tests, not just a review.
  • Backend cost & lock-in. Free-tier limits are generous but finite; keeping the storage seam thin keeps a future move possible.
  • Maintenance sits with a non-coder. Favor dashboard-managed config over anything that needs ongoing hand-editing.
  • Scope creep. Vertex's social layer (Echo, Sacrifice, Vignettes) is tempting to pull forward; the phasing exists to resist that.
